# TeamTrustGate 🛡️

> ИИ-агент для автоматической обработки клиентских запросов от сейлзов в продуктовую команду.  
> Telegram → LLM → Jira. Без бардака, без дублей, без ручной работы.

**Стек:** Python 3.11 · Telegram Bot API · Gemini 2.5 Flash · DeepSeek (резерв) · Jira REST API (v2 + v3) · SQLite · matplotlib · Railway

---

## Зачем это нужно

Сейлзы каждый день получают запросы от клиентов — «хотим фичу X», «не работает Y», «конкуренты уже умеют Z». Раньше это шло в общий чат или в Jira вручную: без структуры, без приоритетов, с дублями и вечным бардаком в бэклоге.

**TeamTrustGate решает это автоматически:**

- Сейлз пишет запрос в Telegram **своими словами** — никаких форм, никакого Jira
- ИИ извлекает суть, оценивает бизнес-риск и приоритет по методологии RICE
- Бот определяет срок исполнения, проверяет на дубликаты и создаёт структурированный тикет в Jira
- Продуктовая команда получает готовый, оценённый запрос и SLA-алерты по застрявшим тикетам
- Сейлз получает уведомление когда статус его тикета изменится

- `_poll_status_changes` — каждые 5 минут проверяет изменения статусов тикетов
- `_check_sla` — раз в сутки проверяет застрявшие тикеты и приближающиеся дедлайны

## Флоу обработки запроса

```
1. Сейлз пишет запрос в Telegram
          │
          ▼
2. LLM анализирует сообщение (extraction.md)
   ├── извлекает суть, клиента, риск, охват, СРОК исполнения
   └── confidence < 0.8? → уточняющий вопрос (до 3 раундов)
          │
          ▼
3. Дедупликация — сравнивает с бэклогом Jira
   └── дубликат найден? → добавляет комментарий, уведомляет сейлза
          │
          ▼
4. RICE-скоринг (scoring.md)
   └── Reach × Impact × Confidence × Strategy Fit → 0–400 баллов
          │
          ▼
5. Превью тикета для подтверждения
   └── [✅ Создать] [✏️ Уточнить] [❌ Отменить]
          │
          ▼
6. Тикет создаётся в Jira (с приоритетом, лейблами и Due Date)
          │
          ▼
7. Фоновые задачи:
   ├── Polling статусов → уведомление при смене статуса
   └── SLA-проверка → алерт продакту если тикет завис / дедлайн близко
```

## Функциональность

### Обработка запросов
- Приём запросов через Telegram обычным языком
- Семантический анализ: очистка от эмоций, паники и сленга продажников
- Извлечение срока исполнения (абсолютные и относительные даты: «к концу квартала» → конкретная дата)
- Уточняющие вопросы при недостатке данных (до 3 раундов)
- Фильтрация спама и нерелевантных запросов (`should_reject`)

### Дедупликация
- Батч-сравнение нового запроса со всеми тикетами за N дней за **один** LLM-вызов
- Семантическое сравнение через LLM (не просто ключевые слова)
- Keyword fallback (Jaccard similarity) при недоступности LLM
- При дубликате — добавление комментария в существующий тикет

### Скоринг (Weighted RICE)
- **Reach** — охват: один клиент / сегмент / все клиенты (1 / 5 / 10)
- **Impact** — риск потери выручки (1–10)
- **Confidence** — уверенность в данных (0.0–1.0)
- **Strategy Fit** — соответствие продуктовой стратегии (1–10)
- **Enterprise-множитель:** если `is_enterprise_risk=true` и `revenue_at_risk≥9` → принудительный подъём reach до 5 и strategy_fit до 10
- Итоговый скор 0–400 → Highest / High / Medium / Low

### Срок исполнения (Due Date)
- LLM извлекает дедлайн из запроса (абсолютный или относительный)
- Валидация: отбрасываются даты в прошлом и невалидные форматы
- Запись в нативное поле Jira `duedate`
- Эмодзи срочности в превью: 🔥 ≤3 дней, ⏰ ≤7 дней, 📅 дальше, ⚠️ просрочен

### Telegram UX
- Превью тикета перед созданием — сейлз видит что распознал ИИ и подтверждает
- Inline-кнопки под каждым тикетом (статус, комментарий, управление для админов)
- Кнопка «💬 Комментарий» — дописать инфу к тикету прямо из Telegram
- Уведомления при изменении статуса тикета (polling каждые 5 минут)
- `/help` — встроенный онбординг с примерами (разный для сейлза и админа)
- Параллельная обработка запросов (`concurrent_updates=True`)

### Ролевая модель
- **Сейлз** (`ALLOWED_USERNAMES`) — создаёт тикеты, смотрит статус, добавляет комментарии, получает уведомления
- **Админ** (`ADMIN_USERNAMES`) — всё вышеперечисленное + управление статусами, удаление, `/stats`, `/export`, SLA-алерты
- **Статусы двигает только продакт-команда** — сейлз создаёт запрос, но не управляет его движением по доске
- Проверка прав на каждом действии (двойная проверка для защиты от устаревших кнопок)

### Jira интеграция
- Создание структурированных Story-тикетов с приоритетом, лейблами и сроком
- Смена статуса через Telegram (▶️ В работу / ✅ Готово / 🔄 На проверку / 🚫 Отклонить) — только админы
- Удаление тикета с двойным подтверждением (только админы)
- Просмотр последних тикетов (`/list`)
- Поиск через новый эндпоинт `/rest/api/3/search/jql` (старый v2/search удалён в Jira Cloud)
- Circuit breaker: 5 ошибок подряд → блокировка на 5 минут

### SLA-алерты (фоновая задача, раз в сутки)
- **Застрявший тикет:** Highest/High висит в To Do дольше N дней → алерт продакту
- **Приближается дедлайн:** Due Date через ≤N дней, а тикет ещё не в работе → алерт
- Каждый алерт отправляется **один раз** (защита от спама через таблицу `sent_alerts`)
- Когда тикет сдвигается в работу — отметка сбрасывается

### Аналитика и экспорт
- `/stats` — статистика за 30 дней **с графиком** (matplotlib): диаграмма приоритетов + круговая по статусам, плюс топ сейлзов
- `/export [дней]` — выгрузка всех тикетов в CSV с BOM для Excel/Google Sheets

### Отказоустойчивость
- Автоматический failover: Gemini → DeepSeek при недоступности
- Retry с экспоненциальным backoff (3 попытки, временные ошибки)
- RAW-тикет при полном отказе ИИ — запрос не теряется
- Сохранение упавших запросов в SQLite для анализа
- Graceful degradation: если график не сгенерился — `/stats` шлёт текст

## Промпты

Все промпты в папке `prompts/` в формате `.md`. Поддерживаются также `.txt` — бот ищет оба формата автоматически.

| Файл | Роль | Назначение |
|------|------|-----------|
| `extraction.md` | Senior Product Analyst | Анализ сырого сообщения, извлечение бизнес-данных и срока исполнения |
| `dedup.md` | Система дедупликации | Сравнение одного запроса с одним тикетом |
| `dedup_batch.md` | Система дедупликации | Один запрос против всего бэклога за один вызов |
| `scoring.md` | Chief Product Officer | Weighted RICE скоринг, расстановка приоритетов |

В `extraction.md` подставляется плейсхолдер `{TODAY}` — текущая дата, чтобы LLM корректно считал относительные сроки.

| `JIRA_EMAIL` | Email аккаунта Jira |

| `JIRA_PROJECT_KEY` | Ключ проекта в Jira (например `KAN`) |

| Переменная | По умолчанию | Описание |
|-----------|:---:|---------|
| `LLM_PROVIDER` | `gemini` | Провайдер: `gemini` / `deepseek` / `openai` / `anthropic` |
| `DEEPSEEK_API_KEY` | — | API ключ DeepSeek (резервный провайдер) |
| `GEMINI_MODEL` | `gemini-2.5-flash` | Название модели Gemini |
| `DEEPSEEK_MODEL` | `deepseek-chat` | Название модели DeepSeek |
| `ALLOWED_USERNAMES` | — | Telegram username сейлзов через запятую (пусто = все) |
| `ADMIN_USERNAMES` | — | Telegram username админов через запятую |
| `PRODUCT_STRATEGY` | — | Описание продуктовой стратегии для скоринга |
| `CONFIDENCE_THRESHOLD` | `0.7` | Порог уверенности для уточнений |
| `DEDUP_DAYS` | `90` | Окно дедупликации в днях |
| `MAX_CLARIFICATION_ROUNDS` | `3` | Максимум раундов уточнений |
| `JIRA_TIMEOUT` | `10` | Таймаут запросов к Jira (секунды) |
| `LLM_TIMEOUT` | `30` | Таймаут запросов к LLM (секунды) |
| `LOG_LEVEL` | `INFO` | Уровень логирования |
| `DB_PATH` | `teamtrustgate.db` | Путь к SQLite базе данных |

### SLA-настройки (в коде, `bot.py`)

| Константа | По умолчанию | Описание |
|-----------|:---:|---------|
| `SLA_CHECK_INTERVAL` | `86400` | Интервал SLA-проверки (раз в сутки) |
| `SLA_STALE_DAYS` | `3` | Через сколько дней застоя Highest/High → алерт |
| `SLA_DEADLINE_WARN_DAYS` | `2` | За сколько дней до дедлайна предупреждать |

## RICE-скоринг

### Формула

```
total_score = Reach × Impact × Confidence × Strategy_Fit
```

Максимум: **400 баллов**

### Компоненты

| Компонент | Значения | Источник |
|-----------|---------|---------|
| Reach | 1 (один клиент) / 5 (сегмент) / 10 (все) | LLM из extraction |
| Impact | 1–10 (риск потери выручки) | LLM из extraction |
| Confidence | 0.0–1.0 (уверенность в данных) | LLM из extraction |
| Strategy Fit | 1–10 (соответствие стратегии) | LLM из scoring |

### Таблица приоритетов

| Балл | Приоритет | Действие |
|------|-----------|---------|
| ≥ 250 | 🔴 Highest | Немедленно в спринт |
| 120–249 | 🟠 High | В ближайший квартал |
| 50–119 | 🟡 Medium | В бэклог |
| < 50 | 🟢 Low | Отложить |

### Enterprise-множитель

Если `is_enterprise_risk = true` **И** `revenue_at_risk ≥ 9`:
- `reach_score` принудительно поднимается до минимум **5**
- `strategy_fit_score` принудительно устанавливается в **10**

## История одного бага: миграция Jira API (HTTP 410)

В процессе разработки Jira Cloud **удалила** старый эндпоинт `/rest/api/2/search`, на котором держался весь поиск. Все команды с поиском (`/list`, `/stats`, `/export`) внезапно стали возвращать пусто, хотя создание и смена статуса работали.

**Диагностика:** добавлена временная команда `/debug`, которая показывала сырой HTTP-ответ Jira прямо в Telegram. Она выявила:

```
HTTP статус: 410
{"errorMessages":["Запрошенный API был удалён.
 Перейдите на API /rest/api/3/search/jql..."]}
```

**Решение:** все поисковые методы мигрированы на новый эндпоинт `/rest/api/3/search/jql`, где параметры передаются в теле POST-запроса (а не в URL). Операции над тикетами (создание, переходы, комментарии) остались на стабильном `/rest/api/2`. Код `410` добавлен в список non-retryable статусов.

Этот кейс — пример работы с реальной деградацией внешнего API в продакшене.
